# Security Toolkit Plugin — Conventions

## Architecture

This is a Claude Code security plugin that orchestrates 50+ tools through skills, commands, MCP servers, Docker containers, and CLI tools. It lives in a monorepo alongside the `opentools` CLI toolkit.

### Monorepo Layout
```
OpenTools/
├── packages/
│   ├── plugin/    — This plugin (skills, commands, config)
│   └── cli/       — Python CLI toolkit (opentools command)
├── engagements/   — Runtime engagement data (gitignored, SQLite-backed)
└── docs/          — Design specs and plans
```

### Directory Structure (packages/plugin/)
```
config/              — Centralized tool and server configuration
  tools.yaml         — Single source of truth for all tool paths
  mcp-servers.yaml   — MCP server connections, health checks, dependencies
  profiles.yaml      — User environment profile (auto-generated by /setup)
shared/              — Shared resources referenced by all skills
  engagement-state.md — Engagement state template (one copy, all skills reference it)
  report-templates/  — Reusable report templates (pentest, incident, cloud, mobile)
skills/              — Skill definitions (SKILL.md + references/)
  pentest/           — Penetration testing workflows
  reverse-engineering/ — Binary analysis and deobfuscation
  hardware-re/       — Firmware, IoT, embedded device analysis
  forensics/         — Digital forensics and incident response
  cloud-security/    — AWS/Azure/GCP security assessment
  mobile/            — Android/iOS application security
commands/            — Slash command definitions
  pentest.md         — /pentest
  reverse.md         — /reverse
  hardware-re.md     — /hardware-re
  vuln-scan.md       — /vuln-scan
  setup.md           — /setup
  recipe.md          — /recipe
recipes.json         — Saved workflow recipes (shareable, reusable)
.env.example         — Template for required API keys and tool paths
```

### CLI Integration

Engagement state is now SQLite-backed and managed via the `opentools` CLI in `packages/cli/`. Skills should invoke `opentools <command> --json` for structured, deterministic tool management rather than reading/writing flat files directly.

Key CLI commands available to skills:

| Command | Purpose |
|---------|---------|
| `opentools engagement create` | Initialize a new engagement (returns engagement ID) |
| `opentools findings list --json` | Query findings with severity/status filters |
| `opentools findings export --format sarif` | Export findings for CI/CD pipelines |
| `opentools preflight --skill <name> --json` | Check tool availability before running a skill |
| `opentools containers start --profile <name>` | Start Docker containers for a skill profile |
| `opentools recipe run <id> --json` | Execute a saved workflow recipe |
| `opentools report generate` | Render a report from templates |

Skills that previously wrote to `engagements/` flat files should now use `opentools engagement` subcommands. The `--json` flag on all commands returns structured output suitable for skill parsing.

### Key Conventions

1. **Tool paths**: Always reference tools via `config/tools.yaml` names or environment variable fallbacks (`${TOOL_PATH:-default}`). Never hardcode absolute paths in skill files.

2. **Engagement state**: One canonical template at `shared/engagement-state.md`. Per-skill references are pointers to the shared file. All skills read/write the same engagement state directory.

3. **Preflight checks**: Every skill starts with a preflight that verifies tool availability before running. Skills degrade gracefully when tools are missing — report what's unavailable and adjust the workflow.

4. **Deduplication**: When multiple tools report the same finding, list all tools as corroboration rather than creating duplicate entries.

5. **Authorization gates**: Active testing, exploitation, and C2 operations require explicit per-action user authorization. Never proceed without confirmation.

6. **Report templates**: Use templates from `shared/report-templates/` for consistent output across engagements.

7. **Skill delegation**: Skills can reference other skills for cross-domain work (e.g., pentest delegates binary analysis to RE, hardware-RE delegates ELF analysis to RE). Note the delegation in the engagement state.

### Adding New Skills

1. Create `skills/<name>/SKILL.md` with frontmatter (name, description, tools)
2. Create `skills/<name>/references/engagement-state.md` pointing to shared
3. Create `skills/<name>/references/tool-paths.md` pointing to shared
4. Add the skill path to `.claude-plugin/plugin.json`
5. Create a command in `commands/<name>.md` if it should be user-invocable
6. Update this file's directory structure
7. Update README.md

### Adding New Recipes

Add entries to `recipes.json` following the schema documented in `commands/recipe.md`. Recipes should specify required tools, variables, steps (sequential or parallel), and output format.
